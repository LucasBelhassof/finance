Observabilidade, health, logs e tratamento de erros

Documenta /api/health, /api/ready, requestId, log estruturado, redacao de dados sensiveis e o error handler global do backend.

requestId por requestHealth e readiness separadosLogs estruturadosRedacao de segredos
Capaz de fazerResponder health/readiness, anexar requestId, registrar conclusao de request e normalizar erros.
LimitacoesNao foi identificado tracing distribuido, APM externo ou dashboard de logs no repositorio.
Permissoes/api/health e /api/ready sao publicos; demais logs/erros sao transversais.
Arquivos relacionadosserver/app.ts, server/shared/logger.js, server/shared/errors.ts.

Visao operacional

Observabilidade no repositório e enxuta e centrada no processo HTTP. Cada request recebe um identificador, logs de conclusao e, em falhas 500, log de erro com contexto sanitizado. Health e readiness foram separados para indicar se a aplicacao esta viva e se o banco esta conectado.

Capaz de fazer

  • Responder /api/health com status ok e horario do servidor.
  • Responder /api/ready com status ready e ping do banco.
  • Propagar ou gerar x-request-id.
  • Logar finalizacao da request com status, duracao e usuario autenticado.
  • Normalizar excecoes e enviar payload padrao de erro.

Nao faz

  • Nao foi identificado tracing distribuido ou correlacao externa alem do requestId.
  • Nao expõe endpoint separado de metrics Prometheus no codigo analisado.
  • Nao registra payload bruto sensivel em logger quando a chave e redigida.

Limitacoes

  • /api/ready depende apenas do ping de banco, nao de todos os servicos externos.
  • Em producao, erros 500 devolvem mensagem generica para o cliente.
  • CSP estrita ainda esta desativada em helmet enquanto frontend/ativos nao forem auditados.

Validacoes

ZodError e erros conhecidos No error handler Normaliza para HTTP coerente 400 validation_error e similares

Erro 500 em producao Error handler Mensagem publica generica The backend failed while processing the request.

Efeitos colaterais

  • Anexa x-request-id em toda resposta.
  • Gera log JSON de conclusao da request.
  • Gera log de erro apenas para falhas 500.
  • Readiness consulta o banco via pingDatabase().
