Como o Finly funciona de ponta a ponta

Esta página resume a arquitetura real do repositório: uma SPA React/Vite com React Query, backend Express, PostgreSQL versionado por migrations SQL, autenticação com sessão e refresh token, fluxos financeiros, IA opcional e uma base operacional leve de health e logs.

React + Vite SPA TypeScript com rotas em src/App.tsx
Express API em server/app.ts e bootstrap em server/server.ts
PostgreSQL Pool pg, migrations em server/migrations/
Observável x-request-id, logs JSON, /api/health e /api/ready

Visão Geral

O Finly é uma aplicação full-stack de finanças pessoais. O frontend renderiza a experiência do usuário, concentra fetches em hooks com React Query e usa src/lib/api.ts como fronteira HTTP. O backend Express aplica segurança, CORS, cookies, autenticação, rate limit e chama a camada de dados em server/database.js, que usa PostgreSQL como fonte persistente.

Regra central

O frontend não recalcula o domínio financeiro pesado quando o backend já entrega dados prontos. Dashboard, parcelamentos, faturas, planos e insights tratam o backend como fonte autoritativa.

Escopo por usuário

As rotas autenticadas usam request.auth.userId. Consultas e mutações de dados financeiros devem permanecer isoladas por usuário.

Incrementalidade

A aplicação já está funcional. Evoluções devem preservar contratos, reutilizar módulos existentes e evitar refactors sem requisito funcional.

Frontend

O frontend é uma SPA React com Vite e TypeScript. src/App.tsx monta os providers globais, as rotas, os guards e o fallback de erro. As rotas nomeadas vivem em src/lib/routes.ts, e as páginas principais ficam em src/pages/.

Composição da aplicação

  • QueryClientProvider disponibiliza cache e mutações do React Query.
  • BrowserRouter controla navegação client-side.
  • AuthProvider faz bootstrap de sessão e configura a camada HTTP.
  • ProductTourProvider controla tour e progresso de onboarding.
  • ErrorBoundary evita tela branca em falhas de renderização.

Fluxo de dados no cliente

  1. 1Página chama hook de feature, como useDashboard ou useTransactions.
  2. 2Hook usa React Query com queryKey estável.
  3. 3Função em src/lib/api.ts faz o request, injeta auth e normaliza payload.
  4. 4Mutações invalidam somente caches afetados, como dashboard, planos e transações.

Backend

O backend é Express. server/server.ts inicializa banco, limpa previews expirados de importação, sobe o servidor e registra shutdown gracioso. server/app.ts monta middlewares, endpoints operacionais, auth, routers modulares e rotas de domínio.

Middlewares globais

A API usa helmet, cors com APP_ORIGIN, cookies, express.json, correlação por x-request-id e logs de conclusão de request.

Proteção de rotas

/api/auth/* é montado antes do guard global. Depois disso, /api passa por requireAccessToken, populando request.auth.

Rate limits

Existem limites específicos para auth, importação, chat e IA, reduzindo abuso em fluxos sensíveis e caros.

O tratamento final de erros converte falhas para JSON com error, requestId e message. Erros 500 são logados com contexto seguro e mensagem genérica em produção.

Banco de Dados

O banco é PostgreSQL. A aplicação usa pg.Pool em server/database.js para a maior parte do domínio e um pool compartilhado em server/shared/db.ts para módulos que foram extraídos. A evolução de schema fica em migrations SQL versionadas.

Bootstrap

  1. 1startServer() chama initializeDatabase().
  2. 2initializeDatabase() executa runMigrations(pool).
  3. 3Migrations aplicadas são registradas em schema_migrations.
  4. 4O servidor sobe somente após inicialização bem-sucedida.

Modelo de domínio

O banco guarda usuários, sessões, contas, cartões, transações, categorias, resumos, insights, mensagens de chat, planejamentos, faturas, habitação, parcelamentos, notificações, previews de importação e eventos de uso de IA.

Migrations antigas não devem ser editadas ou renomeadas. Ajustes novos precisam de nova migration, preferencialmente idempotente, preservando ambientes já aplicados.

Autenticação

A auth é um módulo dedicado em server/modules/auth/ e é consumida pelo frontend em src/modules/auth/. O access token fica em memória no cliente; o refresh token fica em cookie HttpOnly. O frontend sempre envia credentials: "include".

Rotas públicas de auth

POST /api/auth/login, /signup, /refresh, /logout, /forgot-password e /reset-password. As rotas sensíveis têm rate limit.

Rotas autenticadas de conta

GET /api/auth/me, PATCH /api/auth/onboarding, PATCH /api/auth/account, PATCH /api/auth/contact e POST /api/auth/change-password.

Fluxos Funcionais

Dashboard

Consome GET /api/dashboard. O backend monta cards, transações recentes, gastos por categoria, bancos, mensagens e insights, aplicando filtros de período quando enviados.

Transações

CRUD manual em /api/transactions. Mutações invalidam dashboard, listas, insights e planos quando a mudança pode afetar essas superfícies.

Contas e cartões

/api/banks representa contas, cartões e saldos. Configurações de fatura ficam em /api/invoices/cards/:id/settings.

Categorias

/api/categories lista, cria, atualiza e remove categorias com escopo por usuário, preservando categorias em uso.

Parcelamentos

/api/installments/overview entrega a visão analítica pronta: filtros, cards, gráficos, alertas e consistências.

Habitação

/api/housing gerencia despesas fixas e financiamentos. Criação e edição podem gerar ou regenerar transações derivadas.

Faturas

/api/invoices lista períodos de cartão, pagamentos e preferências como fechamento, vencimento e lembretes.

Investimentos e caixinhas

/api/investments controla metas de reserva com aporte fixo ou percentual, podendo se relacionar com planejamentos.

Admin

AdminRoute protege a UI e requireAdminAccess protege o backend. As rotas admin registram auditoria de acesso.

Importação, Chat e IA

Importação

  1. 1Usuário envia arquivo por multipart/form-data.
  2. 2Backend detecta origem e layout, extrai linhas e cria preview.
  3. 3IA pode sugerir categorias e ajustes, sem gravar automaticamente.
  4. 4Commit grava somente itens aceitos e respeita regras de dedupe, contas e parcelas.

Chat e planejamentos

O chat premium usa /api/chats e /api/chats/:chatId/messages. Planejamentos usam /api/plans e podem receber rascunhos, revisões, resumos e avaliações por IA. Eventos de uso de IA alimentam a superfície administrativa /admin/ai-usage.

Senhas de PDF, prompts, tokens, cookies, arquivos importados e payloads financeiros crus não devem ser persistidos em logs. A página documenta o fluxo sem expor exemplos sensíveis.

Health e Observabilidade

Health vs readiness

GET /api/health retorna liveness leve com status e horário do servidor. Não consulta banco. GET /api/ready chama pingDatabase() e confirma conexão com PostgreSQL.

Logs de request

Cada request concluído emite log com requestId, method, path, status, durationMs e userId quando autenticado.

Redação de campos sensíveis

server/shared/logger.js redige chaves contendo fragmentos como password, token, secret, prompt, authorization, cookie, api_key, database_url, statement e extract.

Docker healthcheck

O Dockerfile usa GET http://127.0.0.1:3001/api/health como HEALTHCHECK, evitando depender do banco para liveness do container.

Deploy e Validação Operacional

Subida local

  1. 1npm install instala dependências.
  2. 2npm run db:migrate aplica migrations.
  3. 3npm run server:dev sobe o Express.
  4. 4npm run dev sobe o Vite em outro terminal.

Produção

  1. 1npm run build gera dist e dist-server.
  2. 2npm run server:start executa dist-server/server.js.
  3. 3Container usa Node 22, usuário não-root e porta 3001.
  4. 4Artefato estático da SPA é servido pelo próprio Express quando dist/index.html existe.

Checklist mínimo de release

  • npm run format:check para garantir padrão de formatação.
  • npm run lint para validação estática.
  • npm run test para suites unitárias e integração.
  • npm run build para frontend e backend compilado.
  • npm run test:e2e para smoke Playwright quando aplicável ao release.
  • Verificar /api/health, /api/ready e logs correlacionados por requestId.
