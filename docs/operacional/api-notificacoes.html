API de notificacoes

Mantem a caixa de notificacoes do usuario, incluindo criacao manual pelo proprio usuario, marcacao de leitura e eventos gerados por modulos como faturas, habitacao e planejamento.

Escopo por usuarioInbox propriaAdmin e sistema como origemSem fila externa identificada
Capaz de fazerListar, criar manualmente, marcar como lida/nao lida, marcar todas como lidas e excluir.
LimitacoesNao foi identificada entrega push/e-mail; o dominio e inbox persistida no banco.
PermissoesBearer obrigatorio; acao sempre restrita ao proprio usuario, exceto origem admin em rota separada.
Arquivos relacionadosserver/modules/notifications/*, src/hooks/use-notifications.ts, src/pages/Notifications.tsx.

Visao operacional

O modulo serve como caixa interna do produto. Algumas notificacoes sao criadas por rotinas de negocio, e o usuario tambem pode criar notas proprias. Filtros por status, origem e periodo reduzem a lista retornada.

Capaz de fazer

  • Listar notificacoes do usuario com filtros.
  • Criar notificacao propria com titulo, mensagem, categoria e data opcional.
  • Marcar uma notificacao como lida ou nao lida.
  • Marcar todas como lidas.
  • Excluir notificacao individual.

Nao faz

  • Nao foi identificada paginacao offset/page; a rota usa principalmente limit.
  • Nao foi identificada entrega por e-mail, SMS ou push no repositorio analisado.
  • Nao permite a um usuario comum ler notificacoes de outro usuario.

Limitacoes

  • Titulo e mensagem passam por validacao de tamanho.
  • Data de disparo opcional precisa ser parseavel.
  • Notificacoes administrativas existem, mas sao criadas pela rota admin, nao por esta superficie do usuario.

Endpoints relacionados

Metodo Rota Capacidade Auth Permissao Entrada Resposta Limitacao
GET /api/notifications Lista notificacoes Bearer Usuario autenticado limit, status/unreadOnly, source, startDate, endDate { notifications } Sem paginacao rica identificada
POST /api/notifications Cria notificacao propria Bearer Usuario autenticado title, message, category, triggerAt opcional 201 + notificacao Somente source=user_self
PATCH /api/notifications/:id/read Marca como lida Bearer Usuario autenticado ID notificacao atualizada ID deve ser do proprio usuario
PATCH /api/notifications/:id/unread Marca como nao lida Bearer Usuario autenticado ID notificacao atualizada Escopo do usuario
PATCH /api/notifications/read-all Marca todas como lidas Bearer Usuario autenticado Sem body contagem atualizada Opera apenas no usuario atual
DELETE /api/notifications/:id Remove notificacao Bearer Usuario autenticado ID 204 Sem soft delete identificado

Entradas esperadas

Campo Tipo Obrigatorio Regra Observacao
title string Sim Tamanho maximo validado no service Trim e texto curto
message string Sim Tamanho maximo validado Conteudo principal
category string Nao general, invoice_due, financing_due, installment_due, housing_due, custom Padrao do service
triggerAt data/hora Nao Precisa ser parseavel Pode agendar exibicao

Validacoes

Validacao Quando ocorre Resultado Erro esperado
Titulo e mensagem validos POST Rejeita texto vazio ou longo demais 400
Filtro de datas parseavel GET Bloqueia intervalo invalido 400
Escopo do usuario PATCH/DELETE Impede manipular notificacao alheia 404 recurso nao encontrado

Efeitos colaterais

  • Cria, altera estado de leitura ou remove registros em notifications.
  • Rotas de negocio podem inserir notificacoes com source=system ou source=admin_*.
  • Frontend atualiza cache otimista e invalida counters relacionados.

Erros esperados

HTTP Causa Quando Como corrigir
400 Payload ou filtro invalido Listagem e criacao Corrigir datas e textos
404 Notificacao inexistente PATCH/DELETE Usar ID valido do proprio usuario
401 Bearer ausente Qualquer rota Autenticar
